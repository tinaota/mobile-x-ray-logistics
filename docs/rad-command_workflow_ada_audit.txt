2. ADA / WCAG AUDIT (verified directly against source code)
-------------------------------------------------

DESIGN.md states 48px touch targets and "mandatory high-contrast mode" as requirements, but
implementation doesn't fully deliver on them yet.

[HIGH] Form labels not programmatically associated with inputs — WCAG 1.3.1 / 4.1.2
  Where: client/request/page.tsx, dispatcher/intake/page.tsx, and other form pages
  Pattern found:
    <label className={labelCls}>Patient Name</label>
    <input type="text" ... />
  Findings: 39 <label> tags across the codebase, only 2 htmlFor attributes.
  Impact: Visually fine, but a screen reader focusing the input announces nothing. This affects
  the patient-intake form — the single place accessibility matters most in this product.
  Fix: Add matching id/htmlFor pairs (or wrap the input inside the label element). Low effort,
  high value — applies across ~10 form pages.

[HIGH] STAT priority badge fails color contrast — WCAG 1.4.3
  Where: components/ui/StatusBadge.tsx (PriorityBadge, "stat" variant)
  Pattern found: white text on emergency-red (#EF4444) background — approx. 3.76:1 contrast
  ratio, below the 4.5:1 minimum required for text at this size.
  Impact: This is the badge technicians need to read fastest, often in bright field light
  (per DESIGN.md's own high-ambient-light requirement).
  Fix: Darken the red (e.g. #DC2626 / red-600, ~4.8:1) or keep red-500 as background with
  dark navy text instead of white.

[MEDIUM] Modals have no focus trap or focus return — WCAG 2.4.3
  Where: components/ui/Modal.tsx
  Findings: Escape-to-close works, and role="dialog"/aria-modal are correctly set. However,
  on open, focus is not moved into the panel, and on close, focus is not returned to the
  triggering element. Keyboard/screen-reader users can tab out behind the backdrop.
  Fix: Add a small useFocusTrap hook (~20 lines). Fixes this app-wide since everything routes
  through the one shared Modal component.

[MEDIUM] No live regions for real-time status changes — WCAG 4.1.3
  Where: app-wide (zero aria-live usage found in src/)
  Impact: The client tracking page updates status ("en route" -> "in progress") and ETA in
  place. A screen reader user gets no announcement when this happens.
  Fix: Wrap the status stepper / ETA text in an aria-live="polite" region.

WHAT'S ALREADY SOLID (do not over-fix):
  - Status/priority is never color-only — every badge pairs color with a text label
    (StatusBadge.tsx), correctly satisfying 1.4.1.
  - Icon-only buttons (close, search, notifications, remove) consistently have aria-label.
  - Tabs.tsx, Toast.tsx, and comboboxes use correct ARIA roles (tablist/tab/tabpanel,
    role="alert", role="listbox"/"option").

4. COMPETITIVE COMPARISON — B2B / OPS TOOLING
-------------------------------------------------
Note: RAD-COMMAND is primarily a B2B / operations platform (dispatcher, field technician,
billing manager) rather than a patient-facing app. The comparison below benchmarks against
the facility-side and field-ops tooling of real competitors, not consumer health apps.

Competitors reviewed:
- DispatchExpress (DispatchHealth) — facility-side product embedded inside PointClickCare.
  TridentCare acquired DispatchHealth's imaging business unit (effective Jan 20, 2026),
  which operated portable X-ray/ultrasound across 16 states in home, post-acute, and
  correctional settings.
- Getlabs Portal — booking/orchestration platform used by partner organizations to dispatch
  mobile phlebotomists, track appointment progress, and pull diagnostic data + analytics.
- Generic mobile-phlebotomy dispatch software (category pattern, e.g. TrackoBit-style tools).

Key takeaways:

1. Credentialing is a real differentiator, not a nice-to-have. None of the three competitors
   publicly show anything as built-out as dispatcher/credentials (expiry tracking, license
   numbers, live status states). For a HIPAA-adjacent field-services product this is a
   legitimate selling point in a pitch or interview — most competitors treat credentialing as
   a back-office HR process, not a live dashboard.

2. The embedded-vs-standalone question is the real strategic one, not patient UX polish.
   DispatchExpress's core pitch is that facility staff never learn a new tool — it lives
   inside PointClickCare, which most SNFs already use. RAD-COMMAND is a standalone dispatcher
   desktop. Standalone gives full control over UX (richer fleet/monitoring/reports views),
   but means every new facility has a second system to log into. Worth being deliberate about
   which way to frame this in an interview.

3. No-show/reassignment isn't confirmed to exist in RAD-COMMAND yet. Generic phlebotomy
   dispatch software treats "find a replacement when someone doesn't show" as a named
   feature. Needs verification in dispatcher/assignment — confirm whether this is present or
   a genuine gap.

4. Analytics maturity is close but not equal. Getlabs Portal explicitly sells NPS and patient
   non-adherence rate reporting — a retention/quality layer. RAD-COMMAND's dispatcher/reports
   and billing/reports are operational (response time, completion rate, revenue by facility)
   but lack a satisfaction/retention metric — this maps back to the CRM/retention gap
   identified in Section 1.

[DONE] 2. Fix STAT badge contrast: changed bg-emergency-red -> bg-red-600 in
   StatusBadge.tsx (white text now ~4.8:1, passes WCAG 1.4.3; red-600 already used

[DONE] 3. Add focus trap + focus return to the shared Modal.tsx: focus now moves into the

   returns to the triggering element on close.
[DONE] 4. Add aria-live to status/ETA updates: the hero status card on the client
   tracking page (src/app/client/page.tsx) now has role="status" aria-live="polite" so

   covered — the vertical status stepper elsewhere on the page was not addressed and may
   need the same treatment.)

Implementation: see ada_accessibility_fixes.patch (git diff, 6 files changed) for the full
change set. Apply with `git apply ada_accessibility_fixes.patch` from the repo root.

MEDIUM PRIORITY — real product gaps surfaced by the competitive research
5. Build no-show / reassignment handling in dispatcher/assignment. CONFIRMED MISSING
   (verified via code search — no reassign/no-show/replace logic found). Every generic
   phlebotomy dispatch competitor treats this as a named feature. Functional gap, not a
   polish item — prioritize over anything cosmetic.
6. Add a retention/quality metric to dispatcher/reports or billing/reports (e.g. an NPS or
   facility-satisfaction score). Getlabs Portal sells this explicitly; current reports are
   purely operational. Also closes the CRM/retention gap from Section 1.
7. Stand up a dedicated Results/Interpretation surface, separate from technician/clinical.
   Both the workflow-mapping doc and the competitive set treat "results delivery" as its own
   step; currently folded into capture and thin.

LOWER PRIORITY — worth tracking, less urgent
8. Two-way messaging (already flagged in UX_EVALUATION.md) — chat bubbles currently only show
   outbound messages, not responses. Less urgent given RAD-COMMAND's B2B orientation, but
   still relevant to the dispatcher/facility access-note flow.
9. Decide and document standalone-vs-embedded positioning (vs. DispatchExpress living inside
   PointClickCare). Not a code fix — a product/strategy note worth having ready for
   interviews; explains why fleet/monitoring/reports views can be richer than a competitor's.

NOT WORTH CHASING
Reschedule/cancel self-service windows and SMS-first notifications (Getlabs' strengths) are
patient-facing UX patterns. Given RAD-COMMAND's B2B orientation, deprioritize below everything
above.
